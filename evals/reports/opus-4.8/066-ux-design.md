# Progressive Disclosure vs. Persistent Navigation for AS/400-to-Cloud ERP Migrations: A Synthesis for the Production-Floor Supervisor Persona

## Bottom line

For production-floor supervisors aged 45–60 whose only software mental model is the keyboard-driven AS/400 (IBM i) green screen, the evidence favors **simplified, role-based views that use progressive disclosure for their primary inventory and work-order tasks, with persistent shortcuts to the occasional advanced functions** — rather than a comprehensive dashboard that exposes the full feature set at once. Progressive disclosure is documented to improve three of usability's five components — learnability, efficiency of use, and error rate — precisely the dimensions that determine time-to-competency and error reduction for novice users [1]. All three target platforms already implement role-based view containers as their native pattern — SAP Fiori launchpad (tiles/business catalogs), NetSuite Centers, and Dynamics 365 role-filtered list pages [2][3] — so the design decision is less "role-based vs. dashboard" than *how aggressively to defer advanced functions within the role view, and how to keep the rare advanced action one persistent click away.* The reverse — a comprehensive dashboard — wins only when the "advanced" function is in fact a daily task, because progressive disclosure's benefit collapses when hidden features become everyday ones and users are forced to "menu-dive" for familiar tools [4].

A genuine limitation of this synthesis: the sources retrieved establish the UX mechanics, the legacy mental-model problem, and each platform's role-based architecture firmly, but they do **not** contain vendor-published, quantified usability/training metrics (SUS scores, % task-completion gains, error-reduction percentages) for SAP, NetSuite, or Dynamics inventory/work-order workflows specifically, nor named-manufacturer case-study figures, nor specific cognitive-load-and-aging experimental results or Prosci ADKAR / WalkMe quantified outcomes. Those items are flagged inline below where they belong, so the recommendation rests on the mechanics that *are* documented rather than on figures that are not.

## 1. Progressive disclosure vs. persistent/global navigation: the UX mechanics

**Progressive disclosure**, as framed by Jakob Nielsen (Nielsen Norman Group, Dec 3 2006), "defers advanced or rarely used features to a secondary screen, making applications easier to learn and less error-prone" [1]. Its two-step mechanic: (1) initially show only the few most important options; (2) reveal the larger set of specialized options only on request — the canonical example being a Print dialog with an "advanced options" button [1]. The pattern is implemented through UI devices such as toggles, collapsible sidebars, tooltips/popovers, and modal windows that show and hide advanced settings [4].

Two findings matter for the supervisor persona:
- **It improves three of usability's five components — learnability, efficiency of use, and error rate** [1]. For novice users, the very fact that an option appears on the initial display signals that it is important; this prioritizes attention and helps them *avoid mistakes*, saving the time they would otherwise spend contemplating features they don't need [1].
- NN/g explicitly rebuts the fear that a limited initial view harms users by giving them an impoverished mental model — it calls that fear "groundless," noting people understand a system **better** when helped to prioritize. The practical guardrail is a maximum of **two disclosure levels**; three or more cause users to get lost [1].

**Persistent / global navigation** is the opposite design choice: keeping options always visible. Its strength is **discoverability** (also called learnability) — "the ease at which users can find new features/functions and learn to use them" [4]. Always-visible navigation maximizes discoverability of occasional advanced functions but at the cost of on-screen clutter and scanning effort; progressive disclosure maximizes focus and error-avoidance for the primary task but raises the discoverability cost of the deferred functions [4]. The recommended reconciliation is an **"Action Bar"** approach that lets users "bank" their most-common actions while removing clutter — i.e., surface the frequent, defer the rest [4].

## 2. Effect on task completion and learnability for novice / low-experience users

Documented metrics used to evaluate the pattern are **task completion rate** (higher % suggests the design helps users find what they need), **time on task** (reduction indicates more efficient navigation), **error rate** (lower indicates a more intuitive, less confusing design), and user satisfaction [4]. Progressive disclosure is explicitly aimed at improving usability for **both** novices and experts — showing users what they need when they need it [4].

The central **tension** is documented bluntly: progressive disclosure's benefit **diminishes when "advanced" features become everyday tasks** — if users must "menu-dive" to reach tools they use constantly, the pattern becomes a hindrance rather than a help [4]. Two mitigations are named: (a) redefine which actions count as "primary" via user research, and (b) offer UI customization so frequent-advanced users can pin those functions [4]. A related risk is over-simplification ("dumbed-down content") that strands advanced users [4]. One counterintuitive data point: steeper learning curves can, against standard UX expectation, produce more loyal and capable users *among those who persist* [4] — relevant because the supervisor persona is a captive, persisting user, not a churn-prone consumer.

## 3. The three platforms on the supervisor's inventory and work-order workflows

All three platforms converge on the **same architectural answer** — a role-based view container — but differ in how much they defer versus expose by default. Compared on the same sub-dimensions (role-based container, default density, work-order workflow, inventory surfacing, documented metrics):

| Dimension | SAP S/4HANA (Fiori) | Oracle NetSuite | Microsoft Dynamics 365 SCM |
|---|---|---|---|
| Role-based container | Fiori **launchpad** — a shell giving role-based access to apps; home page of **tiles**, each tile = one business app, with live status indicators (e.g., open-task counts); content scoped by **business catalogs** (tile + target-mapping subsets per business role)  | **Center** — each user sees a different set of tabbed pages based on assigned role; center provides "the pages and links that users with the same role need to do their jobs" [2] | Role-filtered **list pages** (e.g., "My functional location work order maintenance jobs" scoped to the worker's locations) [3] |
| Default information density | Low per tile (one app per tile) — inherently progressive: drill in from a tile to its app  | **Dashboard** of portlets (KPIs, scorecards, trend graphs, workbooks, report snapshots) — comprehensive by default; admin- or self-personalizable [2] | **Dense action pane**: buttons grouped across tabs (New, Edit, Adjust, Schedule, Dispatch, Forecast, Journals, Cost control, Hour control, Work order report, Work order consumption, etc.) — high feature density [3] |
| Work-order / production workflow | Inventory/goods-movement handled by the **Post Goods Movement** app (a SAP GUI-for-HTML transaction, = transaction MIGO), part of the **Warehouse Clerk** (SAP_BR_WAREHOUSE_CLERK) and **Inventory Manager** (SAP_BR_INVENTORY_MANAGER) business roles; the user selects a transaction (Goods receipt / Goods issue / Transfer posting / Place-in/Remove-from storage / Cancel / Display / Change), a reference document (Order, Purchase order, Inbound/Outbound delivery, Material document, Reservation, or "Others" without reference), then enters movement type and special-stock indicator; screen is split into Header data, Item overview, Item details, and Overview tree with info grouped on tab pages [5] | Work-order creation via role-scoped center pages [2]; specific SuiteApp workflow steps not in retrieved sources | Work orders created manually or auto-created (calendar-based maintenance plans/rounds with "Auto create"); managed from "All work orders" list page → details view with Header/Lines tabs and a "Related information" pane [3] |
| Stated design principles | Five Fiori principles: **role-based, adaptive, simple, coherent, delightful**  | Role-driven relevance + real-time data; personalization controllable by admins [2] | Role-filtered views plus comprehensive action pane (discoverability-first within a task) [3] |
| Documented usability/training metrics | Not present in retrieved sources | Not present in retrieved sources | Not present in retrieved sources |

**Interpretation for the supervisor persona.** SAP Fiori's tile model is the most natively progressive: a supervisor sees a small set of role tiles and drills into one app at a time, which aligns with NN/g's "show the few most important options first" prescription [1]. NetSuite's Center is dashboard-first — comprehensive by default, which risks the clutter/scanning cost for a low-experience user, but its admin-publishable, role-scoped dashboards let an implementer pare the default view down to the supervisor's actual tasks [2]. Dynamics' role-filtered list pages help, but its work-order action pane is feature-dense across many tabs [3] — the configuration most exposed to progressive-disclosure's failure mode if not trimmed for the persona. None of the three vendor pages retrieved carries quantified usability or training metrics for these workflows, so cross-platform ranking here is on documented UI architecture, not on measured proficiency outcomes.

## 4. The AS/400 legacy mental model and why it interferes

The supervisor's existing mental model is keyboard-first and deeply ingrained. The IBM i / AS/400 green-on-black "green screen" reflects 30–40-year-old hardware where the keyboard was the **only** input mechanism [6]. Long-time employees become "deeply accustomed" to fixed keystroke sequences — the documented example: "for a particular data entry screen they press **tab 4 times, enter data, tab, enter data, hit enter…done**" [6]. That muscle memory is exactly why "replacement solutions face the added barrier of **requiring substantial training**" [6] — the core source of transition resistance.

The interference is structural: the predictable tab-order, no-mouse efficiency that makes legacy *fast* for veterans is the opposite of the mouse-driven, drill-down GUI/cloud paradigm. Green screens are also documented to require "extra clicks to drill down into key operational details" and offer few integration points [6] — but the relevant point for migration is that a fast keyboard user becomes a slow, error-prone mouse user during transition. Procern's recommendation is to "make user adoption a central" focus, simplify the UI, and surface veterans' "tribal knowledge" as on-screen guidance [6] — which maps directly to a progressive-disclosure-within-role-view design with embedded in-app help.

## 5. Cognitive load theory and how it explains the migration metrics

The mechanics above are explained by cognitive load theory, which distinguishes **intrinsic** load (inherent task complexity), **extraneous** load (load imposed by how information is presented, which does not aid learning), and **germane** load (load that builds schemas/learning). Although the cognitive-load-and-aging experimental study identified in search (van Gerven et al., *Learning and Instruction*, 2002) could not be fetched into the source store in this run, its premise — that efficient training minimizes extraneous load on limited working memory — is the standard reading of the theory and is consistent with the UX evidence retrieved; I flag that the specific worked-example efficiency figures are therefore not grounded here.

The link to interface choice is direct and supported by the grounded sources:
- A **comprehensive dashboard** during transition piles **extraneous** cognitive load onto a user already carrying high **intrinsic** load (new platform, new paradigm, and unlearning the keyboard model [6]). Every irrelevant portlet or action-pane button is scanning cost that does not advance the task [4][2][3].
- **Progressive disclosure** removes that extraneous load by deferring the non-essential, which is *why* it improves error rate and efficiency-of-use for novices [1] — the same two metrics (error rate, time-on-task) used to evaluate it [4]. In change-management terms, lower extraneous load means a shorter climb to competency and fewer errors, because working memory is spent on the task rather than on the interface.

## 6. Metrics frameworks and case-study outcomes

The metrics a migration should track map onto the UX measures above: **task completion rate, time-on-task (→ time-to-competency), error rate (→ error-reduction rate), user adoption rate, and user satisfaction** [4]. Standard named instruments in this space include the System Usability Scale (SUS) for perceived usability, the **Prosci ADKAR** model as an adoption/change-readiness measurement framework, and **digital adoption platforms (DAPs) such as WalkMe** for usage telemetry — and these appeared as named, relevant results in search (Prosci's "Overcome Resistance to ERP Systems Changes With ADKAR"; WalkMe's State of Digital Adoption reporting that only ~30% of companies track digital adoption). I was unable to fetch those pages into the source store in this run, so I state those two figures/frameworks as identified-but-not-grounded rather than citing them as established; likewise, **no named-manufacturer ERP case study with quantified proficiency-gain or error-reduction percentages was successfully retrieved**, so I do not assert specific case figures. The defensible, grounded claim is the metric *set* and its causal tie to interface choice through cognitive load (§5).

## 7. Contextual help, in-app guidance, and digital adoption platforms

Reducing training time during transition is the explicit job of in-app guidance. The grounded recommendation from the legacy-modernization evidence is to surface veterans' "tribal knowledge" as **on-screen guidance and instruction** built into the workflow, rather than relying on documentation alone [6]. This is the function that dedicated DAPs (WalkMe), SAP Enable Now, and Oracle Guided Learning perform for SAP/NetSuite/Dynamics respectively — overlay walkthroughs that lower extraneous load at the moment of need. The retrieved sources establish the *need and mechanism* (on-screen, in-context guidance to replace tribal knowledge [6]); they do not contain quantified training-time-reduction figures for those named platforms, which I therefore do not assert.

## 8. Synthesis verdict: role-based views vs. comprehensive dashboards

**Simplified role-based views (with progressive disclosure) outperform comprehensive dashboards** for this persona when:
- The user's daily work is a small, stable set of tasks (create work order, check/adjust inventory) and advanced functions are genuinely **occasional** [4];
- The user is novice/low-experience and carrying high intrinsic + transition load — deferring the non-essential cuts extraneous load and reduces errors and time-to-competency [1][6];
- Discoverability of the rare advanced function can be preserved by a persistent shortcut rather than by always showing everything [4].

**Comprehensive dashboards win the reverse case** when a supposedly "advanced" function is actually performed frequently — at which point hiding it forces counterproductive menu-diving and progressive disclosure's benefit collapses [4] — or when the user is an expert who scans the full surface faster than they would navigate disclosure steps [4].

## 9. Recommended design: progressive-disclosure-within-role-based-views, with persistent advanced shortcuts

The middle path the evidence points to:
1. **Start from the platform's native role container** — Fiori launchpad business catalog , NetSuite Center [2], or a role-filtered Dynamics list page [3] — scoped to the supervisor role so only the relevant apps/portlets/buttons appear by default.
2. **Apply progressive disclosure within that view**: surface only the few primary inventory and work-order actions first; defer the rest behind at most one disclosure level (NN/g's two-level cap) [1]. On Dynamics specifically, trim or group the dense work-order action pane so the supervisor's New/Edit/Schedule path is foregrounded and Forecast/Cost-control/Journals are deferred [3].
3. **Keep occasional-advanced functions one persistent click away** — an "Action Bar" / pinned-shortcut zone where the rare-but-important functions live, preserving discoverability without cluttering the primary task [4]. Let frequent-advanced users customize/pin, per the documented mitigation [4].
4. **Validate the "primary vs. advanced" split with the actual supervisors** (card sorting / task analysis), because the failure mode is mislabeling a daily task as advanced [4].
5. **Embed in-app guidance** that encodes the veterans' tribal knowledge at the point of need, easing the unlearning of the green-screen tab sequence and lowering extraneous load during the transition window [6][1].
6. **Measure** task completion rate, time-on-task (time-to-competency), and error rate before/after [4] to confirm the disclosure split is helping rather than forcing menu-dives.

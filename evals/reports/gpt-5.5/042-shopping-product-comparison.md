## Comparative assessment

### 1) MFA options and phishing-resistant authentication

All three platforms can meet a GDPR-conscious pharma baseline of mandatory MFA plus stronger authentication for privileged and high-risk access, but they differ in how directly they express phishing resistance in policy.

**Microsoft Entra ID P2 ranks first for policy clarity in Microsoft estates.** Conditional Access has a built-in phishing-resistant MFA strength, and Microsoft’s table marks FIDO2 security keys, Windows Hello for Business/platform credentials, and multifactor certificate-based authentication as satisfying phishing-resistant MFA [5]. Microsoft also lists phishing-resistant methods in Entra ID as Windows Hello for Business, Platform Credential for macOS, synced passkeys/FIDO2, FIDO2 security keys, passkeys in Microsoft Authenticator, and certificate-based authentication; conventional methods such as SMS, voice, OATH, and push are available but are not the preferred phishing-resistant controls [6].

**Okta ranks first for vendor-neutral policy breadth.** Okta Adaptive MFA explicitly lists Okta FastPass, FIDO2 WebAuthn authenticators, and PIV/CAC smart cards as phishing-resistant policies and factors, and its contextual policies can evaluate device, network, location, user behavior, IP address, and device posture [4]. For a company with SAP, Salesforce, laboratory systems, contractors, and mixed endpoints, this is a strong fit when identity policy must sit above multiple application ecosystems rather than inside Microsoft alone.

**PingOne is strong for passwordless and FIDO2.** PingOne supports FIDO2/passkeys in Workforce and Customer environments, including Windows Hello, Mac Touch ID, iOS and Android biometrics, FIDO2 security keys, and non-discoverable credentials [7]. It also supports PingID mobile, a PingID desktop app for Windows/Mac passwordless experiences, third-party authenticator-app TOTP, YubiKey OTP for Workforce, and SMS/voice OTP [7]. PingOne Plus includes passwordless device/token logins including FIDO support at $6/user/month under the 5,000-user-minimum annual model [3].

### 2) PAM capabilities for SOX-style privileged-access evidence

For SOX-relevant controls, the important distinction is whether the platform controls only cloud-directory roles or also records administrative sessions to servers and infrastructure.

**Microsoft Entra ID P2 is strongest for Entra/Azure administrative roles.** PIM provides eligible assignments, activation before use, administrator-configured activation duration, justification, approval workflows, renewal/extension requests, PIM for Groups, and Microsoft Graph APIs for PIM [11]. This directly supports least privilege, segregation of duties, approval evidence, and access-review evidence for Microsoft cloud roles. It is less complete as a standalone PAM replacement for non-Microsoft server sessions unless paired with additional tooling.

**Okta is stronger where PAM evidence must include SSH/RDP sessions.** Okta Privileged Access session recording captures complete SSH and RDP session histories for audit, training, or server monitoring [9]. After sessions end, Okta Privileged Access encrypts and stores session logs, and the client can export, decode, verify, and review logs so they are not tampered with [10]. For SOX control design, that means stronger session-level evidence for infrastructure administrators, although the customer must design retention, storage location, encryption/key management, and review procedures.

**PingOne Privilege is the most ambitious PAM model, but should be evaluated as a separate PAM scope.** PingOne Privilege presents runtime privileged access with zero standing privilege: ephemeral task-scoped access created on demand and revoked at session end, no permanent admin accounts or standing roles, TPM-backed device assurance, policy-based approvals, session recording, continuous audit trails, tamper-resistant logs, and reporting mapped to CIS, HIPAA, ISO, SOC 2, and PCI DSS [12]. It also states that approximately 95% of human privileged access use cases do not require static credentials, reserving vault integration for bootstrap or break-glass scenarios [12]. For SOX, this can be compelling if the company wants a clean-sheet PAM model; commercial and integration scope should be priced separately from PingOne Workforce.

### 3) API rate limits for SAP/Salesforce integration

For SAP and Salesforce, the operational question is whether nightly HR-driven provisioning, SCIM synchronization, entitlement updates, and audit extraction can run without throttling. At 2,500 employees, normal workforce provisioning is unlikely to exhaust Microsoft Graph or PingOne directory limits if integrations use incremental changes, backoff, batching where supported, and separate clients for distinct workloads; Okta needs tenant-specific dashboard validation because Okta exposes consumption through the rate-limit dashboard/System Log and applies endpoint/scoped bucket mechanics.

| Integration-rate point | Okta | Microsoft Entra ID P2 / Graph | PingOne |
|---|---:|---:|---:|
| Directory/user provisioning | Endpoint bucket model; each API token/OAuth app defaults to a **50%** ceiling of the org-wide bucket, so SAP and Salesforce connectors should not share one high-volume principal [13] | For a 2,500-user tenant, identity/access resources allow **8,000 RU/10 sec per app+tenant**; `GET /users` costs **2 RU**, while typical writes cost **1 RU plus 1 write** [15] | Directory read **100/sec**, write **50/sec**, fixed Directory endpoint group **500/sec** [16] |
| Authentication/token calls | Per-username `/api/v1/authn` and `/oauth2/v1/token` are **4/sec** [14] | Global Microsoft Graph cap **130,000 requests/10 sec/app across all tenants** [15] | Token check **100/sec**; SSO API **300/sec**; SSO start **50/sec** [16] |
| Audit/report extraction | System Log and rate-limit dashboard should be used to monitor endpoint-specific quotas and violations [13] | Identity/access reports can be much lower: **5 requests/10 sec per app+tenant**; `signInActivity` GET **10/min** [15] | Audit API **10/sec** [16] |

### 4) EU data residency and GDPR posture

None of the three platforms makes GDPR compliance automatic; each can support a compliant architecture if the company configures EU hosting, DPAs/subprocessors, retention, lawful-basis records, access reviews, breach procedures, and cross-border-transfer controls. For Germany, France, and Poland, EU/EFTA residency and support-access controls should be written into the contract and implementation plan.

**Microsoft Entra ID has the most explicit EU boundary documentation.** For customers with a location in the EU Data Boundary, Microsoft Entra ID stores and processes most customer data in the EU Data Boundary, except for documented temporary, permanent, and optional service transfers [18]. Microsoft describes the EU Data Boundary as covering EU and EFTA countries, with Customer Data and pseudonymized personal data stored and processed, and Professional Services Data stored at rest, in EU/EFTA datacenters for in-scope services [19]. This is the clearest fit for a GDPR review that needs formal boundary language.

**Okta supports customer-selected regional cells.** Okta states that it hosts Customer Data in data centers selected by the customer and has cells in EMEA, along with US, Japan, Australia, Canada, and India [17]. The company should contract for the EMEA cell and align support/subprocessor access controls with its GDPR transfer-impact assessment.

**PingOne supports geography-based hosting.** PingOne organizations are based in a specific geography, and all environment resources for that organization are hosted in that geography; Ping advises having organizations in each geography where the customer operates [20]. Ping’s Data Supplement states that customers select data-residency regions and lists Germany and Ireland among AWS locations, with storage aligned to selected regions for relevant PingOne tenant services [21].

## Ranking for this scenario

1. **Microsoft Entra ID Premium P2 — lowest likely 5-year TCO and best native privileged-role control if Microsoft is already strategic.** Its $1.35M five-year P2 subscription baseline is materially below Okta Essentials and PingOne Plus for this 2,500-user company [1]. It is also the clearest option for Conditional Access phishing-resistant MFA and PIM over Microsoft administrative roles [5] [11]. The trade-off is that SAP/Salesforce and non-Microsoft PAM may require more integration design or complementary tooling.

2. **Okta Workforce Identity Essentials — strongest neutral enterprise IAM package, but higher subscription baseline.** At $2.55M over five years for Essentials, Okta is more expensive than Entra P2 and PingOne Plus on list subscription, but the bundle includes Adaptive MFA, Privileged Access, Lifecycle Management, Access Governance, and 50 Workflows [2]. It is well suited where SAP HR, Salesforce, non-Microsoft SaaS, and heterogeneous endpoint policies are the center of gravity.

3. **PingOne for Workforce Plus with PingOne Privilege — strong security architecture, but commercial fit depends on the 5,000-user minimum and PAM quote.** PingOne Plus’ $1.8M five-year baseline is lower than Okta Essentials but requires paying for 5,000 users even with 2,500 employees [3]. Its FIDO2/passkey support and published rate limits are strong [7] [16], while PingOne Privilege offers a modern zero-standing-privilege model that should be evaluated and priced as a separate privileged-access workstream [12].

**Recommended procurement path:** shortlist **Entra ID P2** as the cost/control baseline, require **Okta Essentials** to justify its higher subscription through faster SAP/Salesforce onboarding and governance/PAM coverage, and keep **PingOne Plus + PingOne Privilege** in contention if passwordless rollout and zero-standing-privilege PAM are strategic enough to offset the 5,000-user minimum. In all three cases, the RFP should require fixed pricing for EU data residency, SAP/Salesforce integrations, migration from AD, MFA/passkey rollout, PAM/SOX evidence design, administrator training, and two post-go-live audit cycles.
